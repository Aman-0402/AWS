Section 01

The Three Main Data Types

Data isn't just rows and columns. It falls into three distinct categories based on how "organized" it is — and choosing the wrong storage for the wrong type is one of the most expensive mistakes a Data Engineer can make.

🗂️1. Structured Data — "Schema-on-Write"

In structured data, you must define the structure before you can save any data. This is called Schema-on-Write.

🚫
The Constraint
If your table has Name and Email columns and you try to upload a row with PhoneNumber too, the database rejects it.
⚡
The Benefit
Because the computer knows exactly where every piece of data lives (e.g., "Email is always Column 2"), it can search millions of rows in milliseconds.
🛠️
AWS Technical Fit
When using Amazon RDS (SQL), you spend time "Normalizing" data — ensuring there is no redundancy across tables.
🔑 Normalization means splitting data into related tables to eliminate duplication. For example, instead of storing a customer's name on every order row, you store it once in a Customers table and reference it by ID.

🌐2. Semi-Structured Data — The King of the Web (JSON)

As a Data Engineer, you will spend roughly 80% of your time dealing with JSON. It is the dominant format for APIs, web events, and mobile apps. 

// A typical JSON event from a web application
{
  "user_id":  101,
  "action":   "click",
  "metadata": {
    "device":     "iPhone",
    "os_version": "17.4"
  }
}
🏷️
Self-Describing
The data includes its own labels (user_id, action). You don't need a separate schema document to understand it.
⚠️
The Flexibility Trap
One JSON object might have a device field; the next one doesn't. This inconsistency is why we call it Schema-on-Read.
⚡
AWS Tool: DynamoDB
A NoSQL database built specifically for this. It doesn't care if User A has more fields than User B — no rigid table structure required.
⚠️ Schema-on-Read explained: With JSON stored in S3, you don't worry about the shape when you store it. You only define and enforce the shape when you query it (via Athena or Glue). This is powerful but can lead to silent data quality issues if you're not careful.

🗃️3. Unstructured Data — The Data Lake Foundation

Unstructured data is the "Heavy Lifting" part of your job. Images, videos, audio files, PDFs — none of these have a queryable structure on their own.

🎬
The Challenge
You can't run a SQL query on a .mp4 video file to find out who the actor is. There's no "column" to search.
🏷️
The Solution: Metadata
You wrap unstructured files in metadata tags. Example: meeting_recording.mp4 → S3 tags: Project: Marketing, Date: 2026-03-10.
🤖
AWS Secret Weapons
Amazon Rekognition (AI vision) and Amazon Transcribe (speech-to-text) convert unstructured media into structured text you can store and query.

Section 03

Data Elements and Metadata

As a Data Engineer, you don't just look at the value (e.g., $50.00). You look at the Data Elements and the Metadata that describe where it came from, what format it is in, and how it should be used.

🧩
Data Elements
The individual pieces of information inside a record.

Examples: Name, Date, Price, UserID, Status
Atomic Units Schema Fields
🏷️
Metadata
"Data about the data." For a photo: the GPS location, the date taken, the file size, the camera model.

In AWS: Glue uses metadata to tell Athena how to read a raw file. Without it, your pipeline is blind.
Glue Catalog S3 Tags
🔍 Real Example: You have 10,000 .jpg images stored in S3. Without metadata, you have a pile of binary blobs. With metadata tags (Category: Product, SKU: ABC-123, Date: 2026-01), you can query S3 directly to find all product images uploaded in January — in seconds.
Section 04

Why This Matters for AWS Architecture

If you try to store Unstructured data (like 10,000 videos) in a Structured database (like RDS), the system will crash or become incredibly expensive. Matching data type to the right service is a core skill.

📐The Data Engineer's Rule of Thumb

🪣
S3 — Everything
Your default landing zone for all data types — especially Unstructured and Semi-Structured. Think of it as the universal inbox.
🏢
RDS / Redshift — Structured
For cleaned, structured business data that needs to be queried quickly with SQL. Redshift is optimized for analytics at scale.
⚡
DynamoDB — Semi-Structured
For Semi-Structured app data (JSON) that needs lightning-fast reads and writes — think user sessions, shopping carts, game states.
If your data is… Store it in… Then analyze with…
Raw logs, videos, images (Unstructured) Amazon S3 Rekognition / Transcribe / Athena
JSON events, API payloads (Semi-Structured) DynamoDB or S3 Athena / Glue / Lambda
Clean sales records, transactions (Structured) RDS or Redshift QuickSight / Direct SQL
Section 05 · Certification Favourite

Why File Formats Matter — The "Big Data" Secret

This is a favourite AWS Certification exam topic. Understanding Row-based vs. Columnar storage can mean the difference between a pipeline that costs $1 and one that costs $1,000 for the same query.

📄
Row-Based Storage
Stores all fields of a record together in sequence. Fast for writing individual transactions, but slow and expensive for analytics queries that only need a few columns from millions of rows.
.CSV .JSON Write-Optimized
📊
Columnar Storage
Stores all values of the same column together. When you only need to calculate "Total Sales," Parquet skips every other column entirely — reading only the data you asked for.
.Parquet .ORC Read-Optimized

💰The Real Cost Difference — A 1TB Example

Format Query: "Calculate Total Sales" Data Read Cost & Speed
.CSV (Row-Based) AWS reads the entire 1 TB file — including all columns you don't need. 1,000 GB 🐢 Slow & Expensive
.Parquet (Columnar) AWS reads only the "Sales" column — skips everything else. ~10 GB ⚡ Fast & Cheap (~100× less data)
🏆 The Gold Standard Rule: Always convert raw CSV/JSON files to Parquet format using AWS Glue before loading them into Athena or Redshift. This single step can reduce your query costs by up to 99%.
Format Type Best Use Case AWS Service Pairing
.CSV Row-Based Writing a single transaction or small export S3, RDS import/export
.JSON Row-Based API payloads, streaming event data Kinesis, DynamoDB, Lambda
.Parquet Columnar Analytics, large-scale aggregation queries Athena, Glue, Redshift Spectrum
.ORC Columnar Hive/EMR workloads, alternative to Parquet EMR, Glue
Section 06

Data Elements: The Atomic Level

When you design a pipeline, you must define the Data Types for each element in your schema. Getting this wrong causes "Data Corruption" — values stored incorrectly that look fine on the surface but produce wrong calculations.

🔢
Integers / Floats
Numbers used for calculations. Use INTEGER for whole numbers (counts) and FLOAT/DECIMAL for prices or measurements.
🔤
Strings / Varchars
Text values: names, addresses, descriptions. A VARCHAR(255) caps the string length — important for storage efficiency.
✅
Booleans
TRUE / FALSE only. Used for flags like is_active_user, is_paid, email_verified.
🕐
Timestamps
Crucial for event ordering. Always store in UTC. Failing to do so causes timezone nightmares when syncing global data across regions.
⏰ The UTC Rule: If your pipeline ingests events from New York (EST), London (GMT), and Tokyo (JST), storing raw local times makes it impossible to sort events in the correct global order. Always convert to UTC (Coordinated Universal Time) at the point of ingestion — before the data ever hits S3.

💥Common Data Corruption Scenarios

Mistake What Goes Wrong Prevention
Storing "$50.00" as a String SUM(price) returns an error or 0 Cast to DECIMAL(10,2) during Glue ETL
Storing dates as "March 10, 2026" Can't sort, filter, or do date arithmetic Normalize to ISO 8601: 2026-03-10T00:00:00Z
Using local time instead of UTC Events appear out of order across time zones Convert to UTC at ingestion (Kinesis / Lambda)
Storing NULL vs. empty string "" COUNT and JOIN queries behave unexpectedly Define explicit NULL handling in your Glue schema
Section 07 · Putting It All Together

Module 04 Strategy: The Master Plan

Combine everything in this module into a single, repeatable strategy. This is the standard data engineering workflow on AWS — from raw chaos to clean, fast, queryable data.

📥 Step 1 · Collect
Ingest everything (Unstructured, Semi-Structured, Structured) into your Amazon S3 Data Lake as raw files. Don't filter anything at this stage — store it all.
▼
🧠 Step 2 · Catalog
Run AWS Glue Crawlers to automatically scan S3 and discover the schema of your files. This populates the Glue Data Catalog — your pipeline's brain.
▼
🔄 Step 3 · Convert
Use AWS Glue ETL jobs to convert raw CSV/JSON files into Parquet columnar format. Fix data types, enforce UTC timestamps, and handle NULLs.
▼
🏢 Step 4 · Load
Load the clean, structured Parquet data into Amazon Redshift for fast SQL analytics, or leave it in S3 and query it cheaply with Amazon Athena.
✅ The Four-Step Mantra: Collect everything into S3 → Catalog with Glue → Convert to Parquet → Load into Redshift / query with Athena. Master this loop and you've mastered the core of AWS Data Engineering.

Q1 · Single Choice
Which schema approach requires you to define the data structure before saving any records?
✅ Schema-on-Write is used by structured databases like Amazon RDS. You must define all columns and types before inserting any data. Schema-on-Read (used by S3 + Athena/Glue) lets you store first and define structure later at query time.
Q2 · Single Choice
You have a 1 TB CSV file and you want to run a query that only needs the "Revenue" column. Which file format would dramatically reduce the cost and time of this query?
✅ Parquet is a columnar format — it stores each column separately, so Athena only reads the "Revenue" column (~10 GB) instead of the entire 1 TB file. This can reduce query cost by up to 99% compared to CSV.
Q3 · Single Choice
Which AWS service is specifically designed to store Semi-Structured JSON data at lightning-fast read/write speeds?
✅ Amazon DynamoDB is a NoSQL database built specifically for Semi-Structured JSON data. It handles variable schemas (different fields per item) and delivers millisecond read/write performance at any scale.
