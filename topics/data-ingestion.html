Section 01

Introduction to Data Ingestion

Data Ingestion is the process of collecting data from different sources and bringing it into a storage or processing system — such as a data lake or data warehouse — where it can be cleaned, transformed, and analysed.

💡 Think of ingestion as the entry door of your pipeline. No data enters the system without passing through it. Every downstream stage — storage, processing, analytics — depends on ingestion being reliable, fast, and complete.

📡Where Data Comes From

🛒
Applications & Databases
E-commerce orders, CRM records, ERP systems, internal business databases — structured transactional data.
🌐
Websites & APIs
Clickstream events, form submissions, REST API responses in JSON or XML — semi-structured data arriving continuously.
📡
IoT Sensors & Devices
Factory machines, weather stations, wearables, connected vehicles — continuous time-series streams with timestamps.
📄
Logs & Files
Server access logs, application error logs, CSV exports, Parquet files — batch files transferred on a schedule.
💳
Payment & Transaction Systems
POS terminals, payment gateways, banking systems — high-frequency transactional data requiring secure handling.
📦
Third-Party & Public Data
SaaS platforms (Salesforce, HubSpot), government datasets, social media feeds, open data portals.

🔄How Pipelines Evolve Over Time

Ingestion pipelines are not built once and left unchanged. As organisations grow, their pipelines evolve to handle more sources, higher volume, and more complex requirements.

Stage Pipeline Design Why It Changes
Early Stage Application → Database → Data Warehouse Simple, few sources, low volume
Growth Stage Multiple sources → Kafka/Kinesis → S3 Data Lake → Warehouse More sources added, volume increases, need for flexibility
Mature Stage Streaming + batch + IoT → Data Lake → Processing → Warehouse → ML Petabyte-scale, real-time needs, ML use cases emerge
💡 Netflix example: Netflix collects user clicks, watch history, device data, and recommendation signals — evolving their pipeline continuously to handle petabytes of data and power the recommendation engine that accounts for 80% of content watched.

ETL vs ELT

Two fundamental approaches exist for moving and preparing data. Understanding when to use each is one of the most important decisions a data engineer makes.

Feature ETL
Extract → Transform → Load		 ELT
Extract → Load → Transform

Raw data preserved No — transformation happens before storage Yes — original data always available for reprocessing

Section 03

Data Wrangling

Data Wrangling (also called Data Munging or Data Preparation) is the process of cleaning, transforming, and organising raw data into a usable format. It is one of the most time-consuming tasks in data engineering — studies show data professionals spend 70–80% of their time on data preparation rather than building models or dashboards.

⚠️ Without data wrangling: dashboards show incorrect results, machine learning models learn from bad examples and produce wrong predictions, and business decisions are made on flawed data.

🛠️Common Wrangling Tools

🐍
Python (Pandas)
Most widely used tool. DataFrames make it easy to clean, reshape, and transform data with code that is readable and reusable.
⚡

Distributed wrangling at scale. When Pandas runs out of memory, Spark distributes the work across a cluster — handles petabytes.
☁️
AWS Glue
Serverless ETL on AWS. Automatically discovers schema, runs PySpark transformations, and writes output to S3 or Redshift.
🗄️
SQL
Powerful for transformation inside data warehouses (Redshift, Athena, BigQuery). CASE statements, window functions, and CTEs handle complex wrangling logic.
Section 04

Step 1 — Data Discovery

Data Discovery is the first step in wrangling — the process of exploring and understanding data before working with it. You cannot clean or transform data you don't understand.

❓Questions Data Discovery Answers

📍
What data exists?
Identify all available datasets — databases, S3 buckets, APIs, file shares, SaaS exports. Use the Glue Data Catalog to browse registered datasets.
📋
What does it look like?
Understand column names, data types, and sample values. Run SELECT * FROM table LIMIT 10 to preview structure.
📏
How big is it?
Understand volume — SELECT COUNT(*) FROM table. Large datasets need distributed processing (EMR/Spark) rather than single-machine tools.
🔗
How are datasets related?
Identify join keys between tables — CustomerID linking orders to customers, ProductID linking orders to inventory.
⚠️
What quality problems exist?
Find missing values, duplicates, and outliers before building pipelines. Use SELECT COUNT(*) WHERE column IS NULL.
✅
Is the data useful?
Determine whether the dataset actually answers the business question. Some data may be available but irrelevant or outdated.

💻Discovery with SQL Queries

-- 1. Preview the structure and sample data
SELECT * FROM customers LIMIT 10;

-- 2. Count total records
SELECT COUNT(*) AS total_records FROM orders;

-- 3. Find missing values in a column
SELECT COUNT(*) AS missing_emails
FROM customers
WHERE email IS NULL;

-- 4. Find duplicate records by key
SELECT order_id, COUNT(*) AS occurrences
FROM orders
GROUP BY order_id
HAVING COUNT(*) > 1;

-- 5. Understand date range of data
SELECT MIN(order_date) AS earliest,
       MAX(order_date) AS latest
FROM orders;

Section 05

Step 2 — Data Structuring

Data Structuring converts raw unorganised data into a structured format that systems can consistently process. Raw data arrives as text files, JSON blobs, CSV exports, or logs — structuring converts these into clean, typed columns in a table.

📄Why Structuring Is Needed

Raw data from different sources rarely has the same format or organisation. A server log, a CSV file, and a JSON API response all look completely different — but they may all contain data that needs to end up in the same analytics table.

Decisions Engineers Make During Structuring Example
What fields to extract from raw data Log line contains 10 fields — extract only date, user, action, country
What data types each column should have price should be decimal, not string; order_date should be DATE, not VARCHAR
How to organise rows and columns Nested JSON objects must be flattened into a single table row
Which schema to use Star schema for analytics warehouse vs flat table for ML features

Modern organisations receive the same type of information from sources in completely different formats. Structuring unifies them.

Section 06

Step 3 — Data Cleaning

Data Cleaning removes errors, inconsistencies, and invalid values from structured data. After data is structured into columns and rows, cleaning fixes what is wrong with the values themselves.

🧹Common Cleaning Tasks

Problem Example Cleaning Action
Duplicate records CustomerID 101 appears twice with identical data Keep one record using DISTINCT or deduplication on primary key
Missing values (NULL) Product price is NULL Replace with default value, median, or flag for manual review; remove row if critical field is missing
Incorrect values Country = "Indai" (typo) Correct to "India" using lookup table or fuzzy matching
Inconsistent date formats "12-03-2026", "2026/03/12", "03-12-26" Standardise to ISO 8601: "2026-03-12"
Wrong casing "aman", "AMAN", "Aman" Apply consistent title case or lowercase across all records
Impossible values Age = -5, Price = -50, Temperature = 5000°C Apply range validation rules; flag or remove invalid rows

🔧Full Cleaning Example

Raw customer dataset with multiple problems:

Customer Email Age
Aman aman@email.com 25
Aman aman@email.com 25
Rahul NULL 30

After cleaning — duplicate removed, missing email filled:

Customer Email Age
Aman aman@email.com 25
Rahul rahul@email.com 30 
-- AWS Athena / Redshift: cleaning with SQL
SELECT
    INITCAP(TRIM(customer_name))     AS customer_name,
    LOWER(email)                       AS email,
    CASE
        WHEN age < 0 OR age > 120 THEN NULL
        ELSE age
    END                                AS age,
    TO_DATE(order_date, 'YYYY-MM-DD')  AS order_date
FROM (
    SELECT *, ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY updated_at DESC) AS rn
    FROM raw_customers
) WHERE rn = 1;   -- keep only the latest record per customer (deduplication)
💡 Important rule: Always retain the original raw data — never overwrite it. Store both the raw version (in the landing/raw zone of your data lake) and the cleaned version (in the trusted zone). If cleaning logic changes, you can reprocess from raw without re-ingesting from the source.
Section 07

Step 4 — Data Enriching

Data Enrichment adds additional useful information to existing data — combining multiple datasets or adding context that makes the data more valuable for analysis.

➕Common Enrichment Tasks

🔗
Combine Datasets
Join orders with customer profiles to add name, city, and segment to each order record — enabling "sales by city" analysis.
🌍
Add External Data
Add public weather data to retail sales data — enabling analysis of how rain affects in-store footfall vs online orders.
🧮
Derive New Fields
Calculate derived columns from existing data — age from birth date, days since last purchase, order value tier (low/medium/high).
📌
Add Missing Attributes
A transaction record may only have CustomerID — enrich by joining to the customer master to add Name, Location, and Segment.

-- SQL enrichment: join orders with customer details
SELECT
    o.order_id,
    c.name         AS customer_name,
    o.product,
    c.city,
    o.amount,
    -- Derived field: classify order size
    CASE
        WHEN o.amount >= 1000 THEN 'High Value'
        WHEN o.amount >= 500  THEN 'Medium Value'
        ELSE                       'Low Value'
    END AS order_tier
FROM orders o
JOIN customers c ON o.customer_id = c.customer_id;
Section 08

Step 5 — Data Validating

Data Validation ensures that the final dataset is accurate, consistent, and follows defined rules before it is published for use. Validation is different from cleaning — cleaning fixes problems, validation confirms the fixes were applied correctly and no new problems were introduced.

📏Types of Validation Checks

Validation Type Rule Invalid Example What Happens
Range Validation Value must be within acceptable range Age = 350 Row flagged as invalid, quarantined for review
Format Validation Value must match expected pattern Email = "aman@email" (missing TLD) Row rejected — email field does not match regex
Type Validation Value must be of the correct data type Price = "twenty" (string, not number) Pipeline fails with type cast error — must be corrected
Relationship Validation Referenced key must exist in another table Order with CustomerID 9999 — no such customer Referential integrity violation — row quarantined
Completeness Validation Required fields must not be NULL Order with no product_id Row rejected — required field missing 
-- Validation query: find rows violating business rules
SELECT
    order_id,
    price,
    customer_id,
    CASE
        WHEN price < 0           THEN 'INVALID: negative price'
        WHEN price > 1000000    THEN 'INVALID: price exceeds maximum'
        WHEN customer_id IS NULL THEN 'INVALID: missing customer'
        ELSE                          'VALID'
    END AS validation_status
FROM orders
WHERE price < 0
   OR price > 1000000
   OR customer_id IS NULL;
⚠️ Best practice: Never silently drop invalid rows. Log every validation failure with the reason, row ID, and timestamp. Invalid data should go to a quarantine or error table so engineers can investigate root causes.

Q 01
Which statement best describes data wrangling?
✅ Data wrangling (also called data munging or data preparation) means cleaning, transforming, and organising data from multiple sources to make it ready for analysis — covering discovery, structuring, cleaning, enriching, validating, and publishing.
Q 02
Which statement about the data validating step is true?
✅ Both data cleaning and data validating involve checking data quality and correctness — they share similarities but occur at different stages. Validating confirms that business rules are met after cleaning; it can be automated and is not limited to any single role.
Q 03
Which task is part of the data publishing step of the data wrangling process?
✅ Data publishing is the final step where prepared data is made available to its consumers. Applying access controls ensures only authorised users can query the data. Fixing outliers belongs to cleaning; field mapping belongs to structuring.
Q 04
A data engineer is designing a pipeline to ingest semistructured data from customer service systems. Which approach should they take?
✅ Data pipelines should be designed based on business requirements and built with flexibility for evolving needs. No single pattern (ETL or ELT) is universally correct — the right choice depends on the use case identified by working with business stakeholders first.
Q 05
A data engineer runs SELECT COUNT(*) FROM orders WHERE customer_id IS NULL. Which data wrangling step is this part of?
✅ This is Data Discovery. The engineer is exploring the dataset to understand what quality problems exist — specifically, how many rows have a missing customer ID. Cleaning would fix them; discovery identifies them first.
Q 06
What is data ingestion?
✅ Data ingestion is the entry point of every pipeline — collecting data from applications, databases, IoT devices, APIs, and files, and bringing it into a central storage system (data lake or warehouse) for downstream processing and analysis.
Q 07
What is the correct order of steps in the data wrangling pipeline?
✅ The correct order is: Discovery (understand what data exists) → Structuring (organise into columns/rows) → Cleaning (fix errors) → Enriching (add more context) → Validating (confirm rules are met) → Publishing (make available to users).
Q 08
What is data wrangling?
✅ Data wrangling (also called data munging or data preparation) covers all the steps that convert messy raw data into clean, structured, analysis-ready datasets: discovery, structuring, cleaning, enriching, validating, and publishing.
Q 09
A validation rule states "price must be greater than 0". An order with price = -50 is found. What should happen?
✅ The correct action is to flag and quarantine the invalid row. Silently deleting data destroys information about what went wrong. Automatically replacing -50 with 0 makes an assumption that may be incorrect. Engineers need to investigate the root cause — was it a refund? A system bug? A data entry error?
Q 10
A data engineer is building a pipeline to ingest unstructured and semi-structured data for exploration by a data scientist. Which ingestion approach is best?
✅ A data lake (Amazon S3) with ELT is the right choice for unstructured and semi-structured data. Raw or near-raw data is loaded immediately, allowing data scientists to explore and transform it later using their own tools and logic. A data warehouse (Redshift) with ETL requires a rigid schema upfront.
Q 11
In the ETL process, in which order do the steps occur?
✅ ETL = Extract → Transform → Load. Data is first collected (Extract), then cleaned and converted to the required format (Transform), and finally stored in the warehouse (Load). In ELT the order differs — Extract → Load → Transform.
Q 12
What is the key advantage of ELT over ETL in modern cloud systems?
✅ ELT's key advantages are speed (raw data loads immediately without waiting for transformation), flexibility (transformation logic can be updated and re-run without re-ingesting data), and the ability to leverage cloud warehouse computing power (Redshift, BigQuery) for massive transformation jobs.
Q 13
A raw CSV file arrives with column names "SUP_ID", "NAME", "COUNTRY". An engineer renames them to "Supplier_ID", "Supplier_Name", "Country" and assigns correct data types. Which wrangling step is this?
✅ This is Data Structuring — organising raw fields into a standardised schema with correct column names and data types. Cleaning fixes values that are wrong; structuring establishes the shape and organisation of the data.
Q 14
An engineer joins an orders table with a customers table to add customer name and city to each order record. Which wrangling step is this?
✅ This is Data Enriching — combining datasets to add new attributes that make the data more valuable. Joining orders with customers adds city information, enabling "sales by city" analysis that was not possible with the orders table alone.
Q 15
Which task might be performed as part of data structuring within the data wrangling process?
✅ Data structuring involves organising and formatting data so it fits the required structure for processing and analysis. Filling in missing data = cleaning. Merging sources = enriching. Choosing visualisation tools = a separate business decision.
Q 16
A food delivery company publishes its wrangled dataset to Amazon Redshift and connects it to a QuickSight dashboard. Which wrangling step has just been completed?
✅ This is Data Publishing — the final step where prepared, validated data is made available to its consumers. Moving data to Redshift and connecting QuickSight makes the data accessible to analysts and managers who need it for business decisions.
Q 17
A business analyst wants to add a new data source for analysing customer interactions. Which data discovery task should be performed first?
✅ The very first discovery task is to evaluate whether the data is worth collecting. If the cost of extraction, cleaning, and maintenance exceeds the business value, the pipeline should not be built at all. Tool selection and storage design come later.
Q 18
Which option describes a task that a data engineer would likely perform during data cleaning?
✅ Data cleaning involves validating and correcting data quality issues such as verifying postal codes, fixing formats, removing duplicates, and handling missing values. Reformatting dates and renaming columns belong to structuring; blindly replacing outliers without investigation is risky practice.
Q 19
Which statement describes a key task in the data enriching step?
✅ Data enrichment means enhancing data by combining it with additional data sources to provide more context and value. Giving users access = publishing. Row counts = discovery. Ingesting data = ingestion, not wrangling.
Q 20
An analyst wants to ingest customer feedback data (emails, text notes, screenshots) for ML. The data is 2 years old and many records have missing customer IDs. What should the data engineer do first?
✅ Before building any pipeline, the engineer should assess whether the data is valuable and usable compared to the effort required to prepare it. Two-year-old data with many missing IDs may not justify the cost. Discovery and value assessment always precede tool selection or infrastructure work.
