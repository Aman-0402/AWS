Section 01

Batch vs Stream Ingestion

One of the most fundamental architecture decisions in data engineering is choosing how data enters the pipeline. The two primary methods are batch ingestion and stream ingestion, and the right choice depends on how quickly the business needs to act on data.

Feature Batch Processing Stream Processing
When data is processed Scheduled intervals — hourly, daily, weekly Continuously — as each event arrives
Data volume per job Large chunks — millions of records at once Small continuous events — one record at a time
Latency High — results available after batch completes Very low — results available within milliseconds
Complexity Simpler to build and maintain More complex — requires always-on infrastructure
Cost Lower — compute runs only when scheduled Higher — infrastructure runs 24/7
Example use case Daily revenue report, monthly invoice generation Credit card fraud detection, live GPS tracking
AWS tools AWS Glue, EMR, DMS, AppFlow Kinesis, Kafka, Lambda, Apache Flink

🎯Choosing the Right Method

Condition Best Method Why
Data comes slowly and in large batches Batch No need to process until a full dataset is ready — cheaper to run once
Data arrives continuously from many sources Streaming Events must be captured immediately — batch would lose the real-time context
Business needs instant decisions (fraud, alerts) Streaming Waiting hours for a batch result would make the decision too late
Historical analysis and reports are the goal Batch All the data must be present to run historical aggregations accurately
Both monthly reports AND real-time dashboard needed Both Use Kinesis Firehose for S3 batch + Apache Flink for real-time dashboard

📦Data Volume and Velocity as Primary Drivers

📦
Volume → Batch
Large datasets — millions of sales records, 10 TB log files — are efficiently processed together in batch. Processing them one by one in real time would be unnecessary and expensive.
⚡
Velocity → Streaming
High-velocity data — thousands of events per second from website clicks or IoT sensors — must be processed continuously. Accumulating it for a daily batch would waste the value of real-time insight.

🗄️AWS Database Migration Service (DMS)

DMS migrates or continuously replicates databases into AWS. It supports both homogeneous (MySQL → MySQL) and heterogeneous (Oracle → Aurora) migrations.

Change Data Capture — DMS continuously replicates ongoing changes from the source database to the target. Source and target stay in sync in near real time.

Section 03

AWS Glue for Batch ETL

AWS Glue is a fully managed, serverless ETL service that automates the most time-consuming parts of building batch data pipelines — schema discovery, job authoring, scheduling, and monitoring — so engineers can focus on data logic instead of infrastructure.

💡 Before AWS Glue: engineers had to manually provision Spark clusters, manage server configurations, monitor cluster health, and write all transformation code from scratch. Glue removes all of that — you just define what you want to do with the data.

🧩Core AWS Glue Components

🕷️
Glue Crawlers
Automatically scan S3 buckets or databases, detect schemas, and register datasets in the Glue Data Catalog. Run once or on a schedule. No schema definition code required.
Schema discovery Auto catalog Zero code
📖
Glue Data Catalog
Central metadata repository — stores table names, column names, data types, and S3 locations. Shared across Athena, Redshift Spectrum, and EMR so all services find data consistently.
Metadata store Table registry Cross-service
⚙️
Glue ETL Jobs
PySpark or Python scripts that clean, transform, and load data. Can be created visually in Glue Studio (drag-and-drop) or written as code. Serverless — no cluster to manage.
PySpark Python Serverless
🎨
Glue Studio
Visual drag-and-drop interface for building ETL jobs. New engineers can create full pipelines without writing code — ideal for getting started quickly.
Visual UI No-code option Quick start
🔄
Glue Workflows
Orchestrate multiple ETL jobs into automated pipelines. Define dependencies: Job A must complete before Job B starts. Run entire pipelines on a schedule.
Orchestration Scheduling Dependencies
🔖
Job Bookmarks
Track which data has already been processed. On the next run, the job only processes new records — not everything from the beginning. Prevents duplicate processing.
Incremental loads State tracking Efficiency

⚙️Glue ETL Job Example

# AWS Glue PySpark ETL job — nightly batch pipeline
from awsglue.context import GlueContext
from pyspark.sql.functions import col, initcap, trim, to_date

glueContext = GlueContext(SparkContext.getOrCreate())

# 1. EXTRACT — read from Glue Data Catalog (maps to S3)
datasource = glueContext.create_dynamic_frame.from_catalog(
    database="sales_db",
    table_name="raw_orders"
)

df = datasource.toDF()

# 2. TRANSFORM — clean and standardise
cleaned = df \
    .filter(col("amount") > 0) \                          # remove invalid amounts
    .dropDuplicates(["order_id"]) \                    # remove duplicates
    .withColumn("customer_name", initcap(trim(col("customer_name")))) \
    .withColumn("order_date", to_date(col("order_date"), "yyyy-MM-dd"))

# 3. LOAD — write Parquet to S3 curated zone
cleaned.write.mode("overwrite").parquet("s3://data-lake/curated/orders/")

📈Scaling and Optimising Glue Jobs

Optimisation Approach How It Helps Example
Horizontal Scaling
Add more workers		 More parallel processing — dataset split across more nodes 1 worker processes 100 GB. 10 workers process the same 100 GB 10x faster.
Vertical Scaling
Larger worker type		 More CPU and RAM per node — handles larger in-memory datasets Standard → G.2X worker doubles memory; good for skewed data or wide rows.
Parquet + Splittable Codec Columnar format enables parallel reads; splittable codec lets multiple workers read one file 100 GB CSV → 20 GB Parquet (Snappy). Queries scan only required columns.
Avoid Too Many Small Files Many tiny files create I/O overhead. Combine into medium-sized files (100 MB – 1 GB). 1 million 1 KB files → 100 files of ~10 MB each. Much faster to process.
Avoid One Giant File One huge file prevents parallel processing — all workers wait for one node Split a 500 GB file into multiple Parquet partitions before processing.
Compression Format Speed Compression Ratio Splittable? Best For
Snappy Very fast Medium With Parquet/ORC Analytics — balance of speed and compression
GZIP Slow High No Cold archive storage — maximum size reduction
Parquet Fast reads High (columnar) Yes Analytics — only read columns you need
ORC Fast High Yes Hive workloads on EMR

📦Kinesis Data Streams — How Records Work

Every record in a Kinesis Data Stream contains exactly three components:

Component Purpose Example
Partition Key Determines which shard the record goes to. Records with the same partition key always go to the same shard — ensuring order per key. "card_id": "4111-1111" — all transactions for this card go to shard 3
Sequence Number Unique identifier assigned by Kinesis when the record arrives. Used to track position within the stream. "49590338271490256608559692540925702759324208523137515522"
Data Blob The actual event payload — up to 1 MB of binary data per record. {"card_id":"4111","amount":850,"location":"Mumbai","ts":"2026-03-12T10:05:00Z"}

📐Scaling Kinesis Data Streams with Shards

A shard is the unit of capacity in Kinesis Data Streams. Each shard supports 1 MB/s write and 2 MB/s read. Adding shards scales the stream horizontally.

Scaling Mode How It Works Best For
Manual Scaling Engineer manually increases or decreases shard count using the console or API Predictable traffic patterns where you know in advance when to scale
On-Demand Mode AWS automatically adjusts shard count based on observed traffic — no manual action Unpredictable or spiky workloads — removes scaling management overhead

Section 05

Ingesting IoT Data by Stream

IoT (Internet of Things) devices — factory sensors, smart cars, fitness trackers, agricultural monitors — generate continuous streams of small events that must be processed in real time. IoT pipelines require specialised protocols and infrastructure designed for low-power devices on constrained networks.

🏭
Industrial Sensors
Machine temperature, vibration, pressure — every second. Anomaly detection prevents equipment failure before it happens.
🌾
Smart Agriculture
Soil moisture, temperature, humidity sensors in fields. When moisture drops below threshold, irrigation is triggered automatically.
🚗
Connected Vehicles
GPS, fuel level, engine diagnostics streamed from a fleet. Enables real-time tracking, predictive maintenance, and route optimisation.
⌚
Wearable Devices
Heart rate, steps, sleep quality from smartwatches. Anomalous readings trigger health alerts to users or medical teams.

📡MQTT — The IoT Communication Protocol

MQTT (Message Queuing Telemetry Transport) is the lightweight messaging protocol used by most IoT devices. It is designed for devices with limited battery and bandwidth — far more efficient than HTTP.

MQTT Concept What It Is Example
Publisher A device that sends data to a topic Temperature sensor publishes {"temp": 28, "unit": "C"} to topic factory/machine-1/temp
Subscriber An application that receives messages from a topic Monitoring system subscribed to factory/machine-1/temp receives every reading
Message Broker The central hub that receives from publishers and distributes to subscribers AWS IoT Core acts as the MQTT broker — no direct device-to-application connection needed
Topic Named channel for messages — hierarchical like a file path home/livingroom/temperature, fleet/truck-42/gps

🔀The Rules Engine — Routing IoT Data

The Rules Engine is what makes AWS IoT Core powerful. It applies logic to incoming MQTT messages and routes them to any AWS service — without writing a consumer application.

# AWS IoT Core Rule example (SQL-like syntax)
# Topic: factory/+/temperature  (+ is wildcard — any machine)

# Rule 1: Store ALL readings in S3 for historical analysis
SELECT machine_id, temperature, timestamp
FROM 'factory/+/temperature'
# Action: Kinesis Firehose → S3

# Rule 2: Alert when temperature exceeds threshold
SELECT machine_id, temperature
FROM 'factory/+/temperature'
WHERE temperature > 40
# Action: AWS Lambda → SNS alert → engineer's phone

Q 01
A data engineer must build a process to ingest sales transactions from a line-of-business database once per day and load them into a data warehouse. Which task would they perform as part of this batch ingestion process?
✅ Batch ingestion requires connecting to the source database to extract records accumulated during the day, then loading them into the warehouse. Real-time producers, Kinesis, and report queries are not batch ingestion steps — they are streaming or consumption activities.
Q 02
A medical research company stores RNA sequencing results in a file store on their on-premises network. Their data science team wants to ingest these files into a data lake on AWS. Which service should they use?
✅ AWS DataSync is designed for transferring large file datasets from on-premises storage (file servers, NAS) to Amazon S3. AppFlow is for SaaS apps, DMS is for relational databases, and Data Exchange is for third-party commercial datasets — none of these fit on-premise file stores.
Q 03
A new ETL developer joins a team and needs to quickly author batch processing jobs. Which AWS Glue feature gives them the fastest start with the least coding required?
✅ AWS Glue Studio provides a visual interface for creating ETL jobs — ideal for new developers. You drag sources, transformations, and destinations onto a canvas and Glue generates the PySpark code. Bookmarks track state, runtime configuration is for performance tuning, and workflows are for orchestrating existing jobs.
Q 04
A data engineer has ingested a new JSON dataset into an S3 bucket. They want data scientists to be able to explore the data immediately with the least amount of coding. Which AWS Glue feature should they use?
✅ Running a Glue Crawler requires zero coding — point it at the S3 bucket, run it, and it automatically detects the JSON schema and creates a table in the Data Catalog. Data scientists can then immediately query the data using Athena or Redshift Spectrum without any additional setup.
Q 05
An AWS Glue job is scheduled to run nightly to update sales reports. The reports have not been updated for 2 days. What is the best first step to troubleshoot the issue?
✅ The first step in troubleshooting any pipeline failure is checking CloudWatch logs and Glue job run insights. These show whether the job ran, whether it failed, and if so, the exact error message. Changing configurations or schedules without understanding the root cause would be premature.
Q 06
An ETL developer wants to optimise the performance of a new AWS Glue job that processes large files. Which approach provides the best performance improvement?
✅ Parquet + Snappy is the optimal combination. Parquet is a columnar format that allows queries to read only required columns — dramatically reducing data scanned. Snappy compression is splittable (with Parquet), enabling multiple workers to read different parts of the same file in parallel. Non-splittable GZIP forces one worker to decompress the entire file sequentially.
Q 07
A data engineer needs to ingest website clickstream data that will be used for monthly reporting AND transformed in real time for a live dashboard. Which Kinesis-based architecture meets both requirements?
✅ Kinesis Data Streams supports multiple simultaneous consumers. Firehose delivers the stream to S3 automatically for batch/monthly reporting. Apache Flink reads the same stream in parallel for real-time analytics. Both consume independently without interfering with each other.
Q 08
Which statement correctly describes how data is stored in Amazon Kinesis Data Streams?
✅ Every Kinesis record has exactly three components: (1) Sequence number — unique ID assigned by Kinesis on arrival; (2) Partition key — determines which shard receives the record; (3) Data blob — the actual event payload up to 1 MB. CloudWatch tracks metrics automatically, so manual scripts are not needed.
Q 09
Users of an application that consumes from a Kinesis Data Stream report periodic gaps in the record sequence. A developer confirms producers are writing correctly and no data is missing. What configuration adjustment is most likely to help?
✅ Gaps in record sequences visible to consumers usually indicate insufficient shard read throughput. Each shard provides 2 MB/s read. If consumers are competing for that capacity, some reads will be throttled causing gaps. Adding shards increases parallel read capacity and resolves consumer throughput issues.
Q 10
A data engineer needs to ingest messages from IoT devices publishing to "lighting" and "heating" MQTT topics, and store lighting data for weekly reporting. Which AWS IoT Core configuration should they use?
✅ The AWS IoT Core Rules Engine is the correct mechanism — it applies SQL-like conditions to incoming MQTT messages and routes matching messages to AWS services. Routing lighting messages to DynamoDB stores them persistently for later weekly reporting. QuickSight cannot directly query MQTT topics; the device registry stores device metadata, not sensor data.
