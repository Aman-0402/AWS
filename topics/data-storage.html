Section 01

Storage in the Modern Data Architecture

After data is ingested and processed, it must be stored in a way that supports easy access, efficient querying, security, and scalability. Modern data architectures use multiple storage layers, each designed for a specific purpose — raw data storage, analytics-ready storage, and operational storage all have different requirements.

💡 Why multiple layers? No single storage system is optimal for every use case. A data lake is cheap and flexible for raw storage; a warehouse is fast and structured for analytics queries; operational databases are optimised for real-time transactions. Using the right tool for each layer saves cost and improves performance.

🏗️Key Storage Layers

Data WarehouseStore cleaned, structured data optimised for analyticsStructured (relational tables)Amazon Redshift

Section 02

Three Types of Cloud Storage

Cloud storage is categorised into three fundamental types based on how data is physically organised and accessed.

Storage TypeHow Data Is OrganisedAWS ServiceBest For
Object StorageFiles stored as individual objects with metadata and a unique keyAmazon S3Data lakes, backups, images, videos, logs — any scale
Block StorageData split into fixed-size blocks — like a physical hard driveAmazon EBSEC2 database volumes, boot disks, high-IOPS transactional workloads
File StorageHierarchical directory tree — folders and filesAmazon EFSShared file systems accessed simultaneously by multiple servers

📦Object Storage — Amazon S3

Amazon S3 is the cornerstone of most AWS data architectures. It stores data as objects in buckets, each with a key, value (the data), and metadata. S3 provides 11 nines of durability (99.999999999%) — designed so data is never lost.

♾️
Virtually Unlimited Scale
No capacity planning needed — S3 scales automatically from kilobytes to exabytes. A company merger doubling your data requires zero infrastructure changes.
🔀
Any Data Format
CSV, JSON, Parquet, images, videos, PDFs, binary files — S3 stores anything without requiring a schema. Ideal for data lakes where format diversity is expected.
💰
Tiered Cost Optimisation
S3 Standard for active data, S3-IA for infrequent access, S3 Glacier for archiving. Lifecycle policies automatically move data to cheaper tiers as it ages.
🔗
Ecosystem Integration
Natively integrated with Athena, Glue, EMR, Redshift Spectrum, SageMaker — every AWS analytics service reads from S3, making it the universal data hub.

💾Block Storage — Amazon EBS

Amazon EBS provides persistent block storage volumes attached to EC2 instances — like adding a hard drive to a virtual machine. EBS volumes persist after the EC2 instance stops, preserving data between restarts.

EBS Volume TypePerformanceBest For

sc1 (Cold HDD)Lowest costInfrequently accessed large datasets — cold storage on EC2

Section 03

Data Lake Storage

A Data Lake is a centralised repository that stores all of an organisation's data — raw, in its original format — at any scale. Unlike a data warehouse which requires data to be structured before loading, a data lake accepts everything first and applies structure later when querying.

CharacteristicData Lake
Schema approachSchema-on-read — structure is applied when the data is queried, not when stored. Raw data is preserved exactly as received.
Data types acceptedStructured (CSV, tables), semi-structured (JSON, XML, Parquet), unstructured (images, videos, logs, audio)
Primary AWS serviceAmazon S3 — unlimited scalability, 11 nines durability, cent-per-GB storage cost
Who uses itData engineers, data scientists, ML engineers — raw data for exploration, model training, ad hoc analysis
Query toolsAmazon Athena (serverless SQL on S3), Redshift Spectrum, EMR/Spark, SageMaker

🏗️Data Lake Zones

🛬
Landing / Raw Zone
Original data exactly as it arrived — never modified. Acts as the source of truth. If transformation logic changes, data can always be reprocessed from here.
s3://lake/raw/ImmutableOriginal format
🧹
Trusted / Cleaned Zone
Data that has been validated and cleaned — duplicates removed, formats standardised, NULL values handled. Stored as Parquet for efficient querying.
s3://lake/trusted/ParquetSchema validated
📊
Curated / Refined Zone
Aggregated, enriched, business-ready datasets. Results of joining multiple tables, calculating KPIs, and preparing data for specific analytical use cases.
s3://lake/curated/AggregatedBI-ready

🏛️AWS Lake Formation

AWS Lake Formation simplifies setting up, securing, and managing a data lake. Instead of manually configuring S3 buckets, Glue crawlers, IAM policies, and data catalog tables separately — Lake Formation provides a unified control plane for everything.

⚡
Quick Setup
Build and secure a data lake in days. Connects directly to RDS (MySQL, PostgreSQL, Oracle, SQL Server) and imports data automatically — no custom pipeline code needed.
🔐
Fine-Grained Access Control
Grant column-level and row-level permissions on data lake tables. A marketing analyst can see customer city but not customer email — controlled at the lake level, not individual S3 bucket policies.
🗂️
Unified Catalog
Automatically registers all datasets in the Glue Data Catalog — shared across Athena, EMR, Redshift Spectrum, and Glue jobs without duplicate configuration.
⚠️ Data lake ≠ data swamp. Without governance, a data lake becomes a "data swamp" — data is dumped in but nobody knows what exists, where it came from, or whether it's trustworthy. Use AWS Lake Formation and the Glue Data Catalog to keep data organised, discoverable, and governed.
Section 04

Data Warehouse Storage

A Data Warehouse stores structured, processed data optimised for analytical queries. Unlike a data lake which stores raw data for exploration, a warehouse is designed to answer business questions fast — daily revenue, top products, regional performance — using SQL at massive scale.

⚖️Data Lake vs Data Warehouse

FeatureData LakeData Warehouse
Data types storedAll types — structured, semi-structured, unstructuredStructured relational data only
Schema approachSchema-on-read — defined at query timeSchema-on-write — defined before data is loaded
Storage costLow — S3 cents-per-GBHigher — compute + columnar storage nodes
Query speedVariable — depends on data format and volumeFast — columnar storage and MPP engine
Primary usersData scientists, ML engineers, exploratory analysisBusiness analysts, BI tools, executive dashboards
AWS serviceAmazon S3 + AthenaAmazon Redshift

🔴Amazon Redshift — AWS Data Warehouse

Amazon Redshift is AWS's fully managed massively parallel processing (MPP) data warehouse. It stores data in columnar format — reading only the columns a query needs — making analytical queries extremely fast even on trillions of rows.

⚡
Columnar Storage
Data stored column-by-column. A query for SUM(revenue) reads only the revenue column — not all other columns in the table. Dramatically reduces I/O for analytics workloads.
🔀
Massively Parallel Processing
Queries are distributed across multiple compute nodes simultaneously. A query that would take minutes on a single machine completes in seconds across 10+ nodes.
🔗
Redshift Spectrum
Query data directly in S3 without loading it into Redshift. Run SQL joins across warehouse tables AND raw data lake files in a single query — bridging lake and warehouse.
🤖
Serverless Option
Redshift Serverless automatically provisions and scales capacity — pay only for queries run. Ideal for intermittent analytics workloads that don't need always-on clusters.
-- Redshift analytics query example — revenue by city and product category
SELECT
    c.city,
    p.category                     AS product_category,
    SUM(o.amount)                   AS total_revenue,
    COUNT(o.order_id)               AS order_count,
    AVG(o.amount)                   AS avg_order_value
FROM orders o
JOIN customers c ON o.customer_id = c.customer_id
JOIN products  p ON o.product_id  = p.product_id
WHERE o.order_date >= '2026-01-01'
GROUP BY c.city, p.category
ORDER BY total_revenue DESC
LIMIT 20;

Characteristic✅ Data Warehouse❌ Not Data Warehouse

Data storedRelational data from transactional systems and operational databasesUnstructured data from IoT, social media, images (that's a data lake)

Section 05

Purpose-Built Databases

Modern applications use specialised databases designed for specific data models and workloads. AWS's philosophy is "use the right tool for the job" — instead of forcing every use case into a relational database, choose the engine that matches your data shape and access pattern.

🎯Three Key Selection Factors

🗂️
Data Shape
Rows and columns (relational), key-value pairs, nested documents, graph nodes and edges, or time-series events — each data shape has an engine designed specifically for it.
⚡
Application Workload
Transactional writes (OLTP), complex analytical reads (OLAP), graph traversal, or time-windowed aggregations — the dominant operation determines the right engine.
📈
Performance Requirements
Sub-millisecond key lookups (DynamoDB), fast analytical scans (Redshift), or consistent ACID transactions (Aurora) — each requires a different storage architecture.

🗄️AWS Purpose-Built Database Catalogue

🗃️

Purpose-built graph database — stores nodes, edges, and relationships. Supports Gremlin and SPARQL queries. Essential for fraud detection graphs and social network analysis.

Section 07

Securing Storage

Data stored in the cloud must be protected against unauthorised access, breaches, and accidental deletion. AWS provides multiple layered security mechanisms for every storage service.

Security LayerMechanismAmazon S3 ExampleAmazon Redshift Example
Encryption at RestAES-256 — data unreadable without the key even if storage is physically accessedS3-SSE with S3-managed or KMS-managed keysCluster-level encryption using AWS KMS — all data files and backups encrypted
Encryption in TransitTLS/HTTPS — data encrypted while moving between services or to clientsEnforce HTTPS-only with S3 bucket policy condition aws:SecureTransportSSL connections required for all JDBC/ODBC connections from BI tools
Access ControlIAM policies define who can read, write, or delete dataS3 bucket policy + IAM role — only specific services/users access specific prefixesIAM roles for cluster access; database users and groups for table-level permissions
Network IsolationVPC restricts which network traffic can reach the storage serviceS3 VPC endpoint — traffic stays on AWS private network, never crosses public internetRedshift cluster inside VPC — only accessible from within the same VPC or via VPN
Audit LoggingRecord every access and change for complianceS3 access logs + CloudTrail — every API call recorded with user, time, and actionRedshift audit logging — all SQL queries, logins, and modifications logged
💡 Banking example: Customer transaction data in Redshift is protected by: encryption at rest with KMS, TLS for all connections, IAM roles granting access only to approved analytics applications, the cluster running inside a private VPC subnet, and CloudTrail recording every query for compliance audits.

Q 01
Due to a company merger, a data engineer needs to increase object storage capacity. They are unsure how much storage will be needed and want a highly scalable service for unstructured, semi-structured, and structured data. Which service is the most cost-effective?
✅ Amazon S3 provides virtually unlimited, highly scalable object storage for all data types at low cost — with no upfront capacity planning. EBS requires fixed volumes; RDS is for relational workloads; Storage Gateway is for hybrid scenarios.
Q 02
A data engineer is evaluating a new database. Which THREE factors should they consider when choosing a database?
Select THREE correct answers
✅ The three key factors are application workload (B), performance requirements (C), and data shape (F). Whether a database is fully managed and data lineage traceability are operational concerns, not primary criteria for selecting the database engine type.
Q 03
A startup is building a web-based order inventory system that requires real-time transactional performance. Which service best meets this need?
✅ Amazon DynamoDB is designed for real-time, high-performance transactional applications — sub-millisecond reads and writes, scales automatically, serverless. Redshift is for analytics, Neptune for graphs, DocumentDB for nested documents.
Q 04
A data engineer needs low-cost infrastructure to query data directly from a central repository containing both structured and unstructured data. Which option meets this need?
✅ A data lake (on Amazon S3) stores structured, semi-structured, and unstructured data at low cost and enables direct serverless SQL querying with Amazon Athena. Redshift handles only structured data and costs more; DMS and QLDB serve different purposes entirely.
Q 05
An organisation is concerned about growing fraud and wants to build fraud detection graphs to analyse relationships between accounts and transactions. Which service is best suited?
✅ Amazon Neptune is designed for storing and analysing relationships — exactly what fraud detection requires. Fraud patterns emerge through multi-hop connections between accounts, devices, and transactions that relational or key-value databases cannot efficiently traverse.
Q 06
A data engineer must host a new MySQL Server database in AWS. Which service should they use?
✅ Amazon Aurora is fully compatible with MySQL, making it a drop-in replacement that runs existing MySQL applications without code changes — while providing up to 5x the performance of standard RDS MySQL. DocumentDB, Neptune, and DynamoDB are all non-relational.
Q 07
Which THREE statements are characteristics of a data warehouse?
Select THREE correct answers
✅ Data warehouse characteristics: schema-on-write (B), stores relational data from operational systems (C), and provides faster queries at higher storage cost vs a data lake (F). Data lakes store all data types including unstructured (rules out A); schema-on-read describes data lakes (rules out D); data lakes have lower — not higher — storage cost (rules out E).
Q 08
A data engineer wants to create a data warehouse in AWS for business intelligence reporting. Which service should they use?
✅ Amazon Redshift is AWS's purpose-built data warehouse — designed for analytical SQL queries using columnar storage and MPP. RDS handles OLTP, DynamoDB handles key-value workloads, and S3 is a data lake storage layer; none are optimised for BI analytics.
Q 09
A data engineer needs a database where data is split across multiple storage nodes and additional nodes can be added as data grows. Which type of database best fits this requirement?
✅ Key-value and wide-column databases (like DynamoDB) are built for horizontal scaling — data is partitioned by key across multiple nodes. As data grows, new nodes are added seamlessly with no downtime. Relational databases primarily scale vertically and have limitations on horizontal partitioning.
Q 10
A data engineer wants to quickly set up a secure data lake and import data from MySQL, PostgreSQL, SQL Server, and Oracle databases in Amazon RDS. Which service should they use?
✅ AWS Lake Formation is the correct answer. It provides a unified service to build, secure, and manage data lakes — with built-in connectors to import from RDS engines including MySQL, PostgreSQL, SQL Server, and Oracle. It handles IAM, encryption, and the Glue Data Catalog automatically. Redshift Spectrum queries existing lake data but doesn't set up the lake itself.
